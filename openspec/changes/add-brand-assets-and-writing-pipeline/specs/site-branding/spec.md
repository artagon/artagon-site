## ADDED Requirements

### Requirement: Static Default OG Image with Per-slug Override

The static `og-image.png` produced by `scripts/generate-logo-pngs.mjs` (1200×630, derived from `brand-svgs.ts`) MUST serve as the site-wide default `og:image` for routes that do not declare a per-slug satori override (the redesign's `site-branding` per-slug OG generation). The two pipelines compose: per-slug satori takes precedence, with the static `og-image.png` as fallback.

#### Scenario: Route without satori override uses static OG

- **WHEN** the `/use-cases` route is built and does not declare a per-slug `cover` for OG
- **THEN** the head contains `<meta property="og:image" content="https://artagon.com/assets/logos/og-image.png">`.

#### Scenario: Route with satori override uses generated OG

- **WHEN** a writing post declares a per-slug satori-generated cover
- **THEN** the head's `og:image` resolves to the satori output, not the static `og-image.png`.

### Requirement: Inline-Glyph Ban (Multi-Surface)

Glyph SVG path data (the Artagon mark, all variants per DESIGN.md §4.14) and wordmark SVG path data MUST NOT appear inline anywhere except `src/data/brand-svgs.ts` and `src/pages/brand.astro` (the gallery route's tile templates). All other consumers (`Header.astro` — pre-pt406 cited as `Nav.astro`; consolidated into `Header.astro` per USMR proposal.md:75 / verified `ls src/components/Header.astro ✓` — `Footer.astro`, MDX, CSS background images) MUST import from `src/data/brand-svgs.ts`. ast-grep rule `rules/security/no-inline-glyph.yaml` MUST enforce the ban across these surfaces:

1. Inline `<svg ... viewBox="0 0 24 24"` in `.astro`, `.mdx`, `.html`.
2. CSS `background-image: url("data:image/svg+xml,...<svg viewBox='0 0 24 24'…")` in `.css` and Astro `<style>` blocks.
3. Astro `<Fragment is:raw>{...}</Fragment>` whose content is a string literal containing `<svg viewBox="0 0`.
4. Other glyph viewBoxes (e.g., 0 0 32 32, 0 0 1024 1024) — the rule MUST match the wordmark and OG-card path data, not just the 24×24 base mark.

`npm run lint:sg:ci` MUST fail on any violation. A documented `note:` field on the YAML rule MUST enumerate the four surfaces.

#### Scenario: Inline glyph in MDX rejected

- **WHEN** a contributor pastes the 24×24 glyph SVG into `src/content/pages/home.mdx`
- **THEN** `npm run lint:sg:ci` exits non-zero with `no-inline-glyph` citing the file:line.

#### Scenario: data: URI in CSS rejected

- **WHEN** a contributor adds `background-image: url("data:image/svg+xml;utf8,<svg viewBox='0 0 24 24'…</svg>");` to a `.css` file
- **THEN** `lint:sg:ci` rejects it with the same rule.

#### Scenario: Import from `brand-svgs.ts` allowed

- **WHEN** `Footer.astro` does `import { SRC } from '../data/brand-svgs'` and renders the SVG via component composition (a `<BrandGlyph>` wrapper component that returns the SVG factory output as Astro JSX, avoiding the banned `set:html` directive)
- **THEN** the lint passes; ast-grep `no-set-html-directive` is not violated.

### Requirement: Brand SVG Factory Cannot Fetch External Resources

`src/data/brand-svgs.ts` SVG factory output MUST NOT contain `<image href="...">`, `<image xlink:href="...">`, `<use href="http...">`, `@import url(http...)`, or any other element that causes sharp/librsvg to fetch external resources at PNG generation time. ast-grep rule `rules/security/no-external-href-in-svg.yaml` MUST enforce this on `src/data/brand-svgs.ts`. The same rule MUST apply to the `/brand` gallery's "Copy SVG" payload — clipboard SVGs MUST be self-contained.

#### Scenario: External href in factory output fails CI

- **WHEN** a contributor adds `<image href="https://attacker.example/img.png"/>` to a factory in `brand-svgs.ts`
- **THEN** `lint:sg:ci` exits non-zero before sharp ever processes the SVG.

#### Scenario: PNG generation makes no outbound calls

- **WHEN** `npm run build:logos` runs in CI under `--network=none`
- **THEN** the script completes successfully (factory outputs are self-contained, no fetch attempts).

### Requirement: Deterministic Favicon Generation

Favicons MUST be generated by `scripts/generate-favicons.mjs` (sharp-based) from `src/data/brand-svgs.ts` source. The generator MUST emit `public/favicon.ico` (multi-size: 16, 32, 48), `public/favicon.svg` (using `currentColor` for theme inheritance), `public/apple-touch-icon.png` (180), `public/icon-192.png`, `public/icon-512.png`, and `public/mask-icon.svg`. The generator MUST run as part of `npm run prebuild` (BEFORE `astro build` copies `public/` into `dist/`); wiring it into `postbuild` would produce stale deployed artifacts. CI MUST run `npm run build:favicons && git diff --exit-code` and fail on any divergence between source and committed outputs.

#### Scenario: Favicon outputs match source

- **WHEN** CI runs `npm run build:favicons` followed by `git diff --exit-code` on the favicon paths
- **THEN** the diff is empty and the check passes.

#### Scenario: Hand-edited favicon fails CI

- **WHEN** a contributor edits `public/favicon.svg` directly without changing `brand-svgs.ts`
- **THEN** the regenerated output differs and CI fails with the offending path.

### Requirement: Deterministic Logo PNG Generation

Logo PNGs MUST be generated by `scripts/generate-logo-pngs.mjs` (sharp-based) from `src/data/brand-svgs.ts` source. The generator MUST emit the 15 outputs documented in `public/assets/logos/README.md`: `logo-mark-{32,64,128,256,512}.png`, `logo-square-{512,1200}.png`, `logo-wordmark-{200,400,800}.png`, `logo-full-{400,800,1200}.png`, `logo-horizontal-800.png`, `og-image.png`. The generator MUST run as part of `npm run prebuild` (BEFORE `astro build`); postbuild generation would not reach `dist/`. CI MUST run `npm run build:logos && git diff --exit-code public/assets/logos/` and fail on any divergence.

#### Scenario: Drift fails CI

- **WHEN** `brand-svgs.ts` changes a glyph path but the committed PNGs are not regenerated
- **THEN** `git diff --exit-code public/assets/logos/` after `npm run build:logos` is non-empty and CI fails.

### Requirement: Glyph Parity with DESIGN.md (Coordinated-Update Gate)

`scripts/check-glyph-parity.mjs` (`npm run check:glyph-parity`) MUST diff `src/data/brand-svgs.ts` SVG path data against `tests/fixtures/brand-glyph-snapshot.json`. The snapshot is derived from DESIGN.md §4.14's canonical viewBox/path constants. Any drift MUST fail the build; updating the snapshot MUST require a coordinated DESIGN.md change verified by a CI gate: when `tests/fixtures/brand-glyph-snapshot.json` is part of a PR's diff, `DESIGN.md` MUST also be in the same PR's diff (otherwise CI fails with a "snapshot regenerated without DESIGN.md update" error). The gate uses `git diff --name-only origin/main...HEAD` to detect both files.

#### Scenario: TS module diverges from DESIGN.md fails

- **WHEN** a contributor edits the classic-glyph triangle path in `brand-svgs.ts` without updating the snapshot or DESIGN.md
- **THEN** `npm run check:glyph-parity` exits non-zero citing the offending variant and the diff.

#### Scenario: Snapshot regenerated without DESIGN.md update fails

- **WHEN** a PR contains changes to `tests/fixtures/brand-glyph-snapshot.json` but no changes to `DESIGN.md`
- **THEN** the parity-coordination CI gate exits non-zero with a "snapshot regenerated without DESIGN.md update" error.

#### Scenario: Coordinated update passes

- **WHEN** a PR contains coordinated changes to BOTH `tests/fixtures/brand-glyph-snapshot.json` AND `DESIGN.md`
- **THEN** both `check:glyph-parity` (TS-vs-snapshot) and the coordination gate pass.
