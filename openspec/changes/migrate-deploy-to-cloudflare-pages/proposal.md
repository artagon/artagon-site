## Why

The site is hosted on **GitHub Pages** today (`actions/deploy-pages`, CNAME → `artagon.com`). USMR's `_redirects` file format and `update-site-marketing-redesign` Phase 7-10 work assume a **Cloudflare Pages-compatible host** — see `openspec/changes/update-site-marketing-redesign/specs/site-bridge-story/spec.md:47` (case-insensitive `/bridge` → `/platform` 301 via `public/_redirects`) and `update-site-marketing-redesign/tasks.md` Phase 10.5. GitHub Pages does not support `_redirects`, server-side redirects with case-insensitive matching, custom HTTP headers, or origin protection. This change migrates the deploy target to **Cloudflare Pages** so the redesign's redirect contract becomes implementable, and tightens the security posture along the way.

Concrete gaps with the current GH-Pages setup:

- **No custom HTTP security headers** (HSTS, COOP, COEP, Permissions-Policy, Report-To, NEL, X-Content-Type-Options, etc.). Today CSP ships only as a `<meta http-equiv>` tag in HTML — fine for a single layer, but headers are the canonical surface.
- **No origin lock-down**: the GitHub Pages origin (`<user>.github.io`) is publicly addressable without proxy; CSP/HSTS settings cannot prevent direct origin access.
- **No edge-caching control**: cache-busting is achieved only via Astro's hashed asset filenames; no header-level `Cache-Control: public, max-age=31536000, immutable` for hashed assets vs `no-cache` for HTML.
- **No request-routing rules**: `_redirects` and `_headers` are unimplementable.
- **No DDoS / WAF protection**: GH Pages relies on GitHub's anycast network without per-route rules.
- **No bot challenge**: scrapers and AI crawlers are uncontrolled.

Cloudflare Pages provides headers, redirects, edge cache rules, free WAF, anycast, and free unmetered bandwidth. The migration is mechanical (push the same `.build/dist` to a Cloudflare Pages project) but the security architecture deserves explicit design — particularly the CSP header transition, HSTS, COOP/COEP for cross-origin isolation, and the rollback strategy if Cloudflare proves unsuitable.

## What Changes

- **Add `dist/_headers` (generated by `scripts/csp.mjs`)** — Cloudflare Pages config for HTTP response headers per path. Sets HSTS (`max-age=300; includeSubDomains` on day 0 per staged rollout — see Staged HSTS Rollout requirement), CSP-as-header (union of all per-page `<meta>` CSPs — see D3), Permissions-Policy (deny camera/microphone/geo/etc. by default), Referrer-Policy `strict-origin-when-cross-origin`, X-Content-Type-Options nosniff, COOP same-origin, COEP credentialless. `public/_headers` is gitignored; the artifact is written to `dist/_headers` by csp.mjs during postbuild to avoid Astro's public→dist copy-ordering trap.
- **Add `public/_redirects`** — same-origin-only redirects (USMR's `/bridge` → `/platform` 301; case-insensitive variants). Validated by `scripts/validate-indexation.mjs` (USMR-defined; reused).
- **Cache headers per file class** in `_headers`:
  - `/assets/*` (hashed by Astro): `Cache-Control: public, max-age=31536000, immutable`
  - `/assets/fonts/*`: same as above + `Access-Control-Allow-Origin: *` (so the WOFF2 still loads if a future change embeds an iframe; redundant under same-origin but cheap insurance).
  - `*.html`: `Cache-Control: public, max-age=0, must-revalidate` + `Vary: Accept-Encoding`.
  - `/sitemap-index.xml`, `/sitemap-0.xml`, `/robots.txt`: `Cache-Control: public, max-age=3600`.
- **GitHub Actions workflow swap**: replace `.github/workflows/deploy.yml` (currently uses `actions/deploy-pages`) with a new Cloudflare Pages deploy via `cloudflare/wrangler-action@v3` using an API token stored in `secrets.CLOUDFLARE_API_TOKEN`. Cloudflare's `Account.Cloudflare Pages:Edit` permission scopes at the ACCOUNT level (not per-project) — see design.md D5. The security boundary is therefore a **dedicated Cloudflare account** that holds ONLY the artagon Pages project; the token's account-resource filter then effectively achieves single-project scope because the account contains only one project. Per multi-reviewer-r1 finding [CR-002].
- **DNS migration plan**: cutover playbook for `artagon.com` apex CNAME from GitHub Pages → Cloudflare Pages. Document TTL drop window (5min), pre-cutover verification (Cloudflare project healthy at preview URL), rollback (revert CNAME). Coordinate the cutover with a `workflow_dispatch` deploy to ensure both targets serve identical content during DNS propagation.
- **Origin protection**: enable Cloudflare's "Always Use HTTPS" rule + "Automatic HTTPS Rewrites" + minimum TLS 1.2. Disable Cloudflare's auto-minify (it would break Astro's SRI hashes).
- **Build-time CSP/header generation**: extend `scripts/csp.mjs` to ALSO emit `dist/_headers` body (NOT `public/_headers`; see design.md D3), so the same source-of-truth (CSP directives) drives both the meta tag (HTML belt-and-braces) and the HTTP header (Cloudflare). Remove drift between the two. `public/_headers` is gitignored.
- **Lighthouse CI gate update**: `lighthouserc.json` baseline currently runs against the GH-Pages-hosted preview; switch to the Cloudflare Pages preview URL once cutover is staged.
- **Documentation**: add `docs/deploy.md` with the cutover playbook + rollback procedure + monitoring expectations.

- `cloudflare-pages-deployment`: governs the Cloudflare Pages contract — `_headers` shape, `_redirects` shape, the Wrangler-based deploy workflow, the API-token scope, edge-cache rules per file class, the origin-lockdown settings, the DNS cutover playbook, and the rollback to GH Pages.

### Modified Capabilities

- `github-pages-deployment` (existing): MARKED for archive once Cloudflare cutover succeeds. Until then, BOTH workflows must succeed on every PR (parallel deploy lets us verify content parity). After cutover, the GH-Pages workflow archives via a separate change.
- `check-site-quality`: ADDED requirement "Cloudflare Pages Headers Verification Gate" — `scripts/verify-headers.mjs` MUST run as part of postbuild; fails if `dist/_headers` declares a `/*` CSP that is not a superset of the per-page `<meta>` CSPs emitted by `scripts/csp.mjs`.

## Out of Scope

- **Cloudflare Workers** — only Cloudflare Pages (static hosting). Rationale: Pages is sufficient for this site; Workers adds a runtime layer we don't need. If future requirements need server-side logic, that's a follow-up OpenSpec change.
- **Cloudflare Functions / Edge Functions** — same reasoning. Rationale: keep the deploy artifact a pure static `dist/`.
- **Cloudflare R2 for asset hosting** — assets ship inside `dist/`. Rationale: Cloudflare Pages serves them from edge cache, no need for a second storage tier.
- **Cloudflare Turnstile / WAF custom rules** — default Pages security level (Medium) + Always Use HTTPS + minimum TLS 1.2 is sufficient for round 1. Rationale: the site has no auth, no forms, no user-generated content; aggressive WAF/captcha would block legitimate bots (Lighthouse, Playwright crawlers) without security benefit.
- **Self-host-woff2-fonts work** — separate OpenSpec change. Rationale: host-agnostic.
- **Any change to `<meta http-equiv>` CSP emission** — keep emitting it in HTML as belt-and-braces. Rationale: the meta tag is a redundant safety net if `_headers` is misconfigured or the host fails to honor the header.
- **Cloudflare Analytics / Web Analytics opt-in** — defer to a privacy-policy-aware follow-up. Rationale: privacy considerations + EU consent surface deserve their own design.
- **Multi-region failover** — single Cloudflare project, single account. Rationale: anycast already provides global resilience; multi-region is over-engineering at this scale.
- **GH-Pages workflow ARCHIVAL** — this change KEEPS both workflows running in parallel. A separate follow-up OpenSpec change archives the GH-Pages workflow once cutover is verified stable for ≥ 7 days. Rationale: deliberate two-step lets us roll back via DNS without code changes.
